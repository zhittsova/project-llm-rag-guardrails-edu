This deck is the RAG-focused version of the milestone presentation.

The point is not that the assistant is production ready. The point is that we have a complete local RAG loop, we can inspect it, and we can compare it with and without guardrails.

The technical story is corpus preparation, chunking, embeddings, vector index, retrieval, answer generation, citations, and then the guardrails around the RAG flow.

RAG architecture

End-to-end flow

The pipeline is deliberately local and inspectable, so every artifact can be shown in the repo.

CorpusJSONL course material
Chunkingsplit documents into retrievable units
Embeddingturn chunk text into vectors
Chromapersist vectors and metadata
Retrievalfind closest chunks for question
Answerextractive response plus citations

Current implementation

data/python_course_docs.jsonl -> langchain_chunk_documents(chunk_size=650, chunk_overlap=80) -> HashingEmbedder(dimensions=384) -> Chroma collection: course_chunks -> VectorRetriever.search(top_k=3) -> answer + citations + retrieved_chunk_ids

This is the main technical slide. The user question does not go straight to an LLM. It goes through retrieval first.

Our corpus is normalized JSONL. Documents are split into chunks. Each chunk gets an embedding. Chroma stores the embedding, the chunk text, and metadata like doc id, course id, title, visibility, and source type.

At query time, we embed the question, search the vector store, and build an answer from the retrieved chunks.

Corpus preparation

The corpus contract is the handoff

Workshop 2 depends on a normalized corpus, not raw PDFs or loose folders.

Required document fields

{ "doc_id": "lec01", "course_id": "python-intro", "title": "6.100L Lecture 1", "visibility": "public", "source_type": "lecture_note", "text": "..." }

Why this matters

  • course_id lets us select the correct course.
  • visibility lets guardrails block private retrieval.
  • title and doc_id become citations.
  • source_type separates lectures, policies, assignments, and admin notes.
validated JSONL metadata preserved citation ready

This slide explains why the corpus work is not just file cleanup. The metadata becomes part of the RAG behavior.

Visibility is especially important for guardrails. If a document is marked private, the guardrailed retriever should not use it for student-facing answers.

The live Python corpus is used for normal course Q and A. The synthetic corpus is used to test private retrieval and injected content.

Chunking strategy

We retrieve chunks, not whole lectures

Chunking makes long course documents searchable and keeps answers tied to smaller evidence units.

Main vector chunking

  • LangChain recursive splitter
  • chunk_size=650 characters
  • chunk_overlap=80 characters
  • chunk id format: lec01:lc:3
chunk 1
80 overlap
chunk 2

Why this is good enough now

  • Definitions are less likely to be cut off.
  • Chunks keep citations closer to the actual evidence.
  • It is deterministic and easy to explain.
  • It works for the local vector demo.
Known limitation: chunking is not yet Markdown-section-aware or slide/page-aware.

This is the chunking slide. The important phrase is: we retrieve chunks, not full documents.

For vector retrieval, we use LangChain's recursive text splitter with 650 characters and 80 characters of overlap.

The overlap is there because important definitions often sit near boundaries. Without overlap, retrieval can lose context.

The limitation is that it does not yet preserve every Markdown section or page number as first-class citation metadata.

This slide explains the vector store. We use Chroma as a persistent local vector store.

The embedding function is intentionally simple: local hashing embeddings. That means no API key, no downloads, no network dependency, and stable results.

The tradeoff is retrieval quality. This is not a production semantic embedding model, but it is enough to show the architecture and failure analysis.

Baseline RAG

The baseline is useful because it is unsafe

It gives us a comparison point: same retrieval path, no first guardrails.

Baseline path

question
retrieve all
top chunks
extract answer
citations

No safety checks

  • no prompt-injection detection
  • no PII or sensitive-data blocking
  • no visibility=public filter
  • no context sanitization
  • no output guard
private retrieval risk injection risk

The baseline is not the model we want to deploy. It is the model we compare against.

Baseline RAG retrieves from the index and builds an extractive answer. It does not filter private documents, it does not detect prompt injection, and it does not sanitize retrieved context.

That is exactly why it is useful for failure analysis.

Guardrailed RAG

First guardrails wrap the RAG path

The guardrails are simple, but they target the first milestone risks.

input guardinjection, PII, unsafe
course filterselected course only
visibility filterpublic docs only
sanitize contextretrieved text is untrusted
answerextractive + cited
output guardgrounding, leakage, integrity

Core retrieval rule

guardrailed mode: course_id = selected course allowed_visibility = {"public"} retrieved context = sanitized before answer generation

The guardrailed version wraps the RAG path with checks before retrieval, during retrieval, after retrieval, and after answer generation.

The key retrieval rule is public-only retrieval for the selected course.

Also, retrieved text is treated as untrusted. This matters for indirect prompt injection: an instruction inside a retrieved document should not become an instruction to the assistant.

Failure analysis

What slips through without guardrails

The synthetic corpus gives us controlled failure cases that the live public corpus cannot show alone.

Baseline failures

  • retrieves private-roster for sensitive-data question
  • repeats injected retrieved text
  • does not enforce academic-integrity behavior
  • does not abstain reliably on unsupported questions
Baseline vector: 3 / 12

Guardrailed behavior

  • blocks sensitive-data request before retrieval
  • filters retrieval to public chunks
  • sanitizes injected context
  • uses citation/abstention rule
Guardrailed vector: 12 / 12

This slide is where the milestone gets persuasive.

The baseline can answer normal questions, but the same mechanism retrieves unsafe or inappropriate context when asked adversarial questions.

The guardrailed version is not magic. It is simple rules and filters. But those rules directly address the first failure modes.

Limits and next work

Ready for the milestone, not production

The honest story: the architecture works, the failure contrast is visible, and the next improvements are clear.

Current limitations

  • local hashing embeddings, not production semantic embeddings
  • extractive answer generator, not a real LLM adapter
  • rule-based guardrails
  • small evaluation set
  • public-only live Python corpus
  • chunking is not yet section/page-aware

Next work

  • stronger embedding model
  • optional LLM answer generator
  • Markdown-section-aware chunking
  • richer citation metadata
  • larger adversarial eval set
  • better demo UI for side-by-side comparison

This is the closing slide. I would be explicit that this is ready for the milestone, not for deployment.

The strongest claim is not production quality. The strongest claim is that the RAG architecture is working end to end and the first guardrails visibly change the failure behavior.

The next work is straightforward: better embeddings, real LLM adapter, smarter chunking, better citations, and a larger red-team style evaluation set.
