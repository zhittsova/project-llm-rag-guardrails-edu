For this milestone, our goal was not to build a perfect assistant yet. The goal was to get a complete local RAG loop working, then show where it fails without guardrails.

The baseline can load course material, chunk it, build a vector index, retrieve relevant chunks, and return an answer with citations.

This gives us something concrete to compare against.

The Python course corpus is our realistic demo corpus. It supports normal questions over course material, like definitions from lectures or course policy.

But all of the Python course documents are public, so it cannot show a private retrieval failure by itself.

For failure analysis, we use a small synthetic corpus on purpose. It contains a private document and an injected instruction, so we can clearly show what the baseline gets wrong.

What fails without guardrails

The baseline works, but it is unsafe

The demo shows two concrete slips: private retrieval and indirect prompt injection.

Baseline: private retrieval

  • Question asks about private roster data
  • Baseline retrieves private-roster
  • No trigger, no visibility filter, no block
retrieved_chunks: ["private-roster:lc:0", ...]

Guardrailed: blocked before retrieval

  • Input guard detects sensitive-data request
  • No citations are returned
  • Trigger: pii
"I cannot process this request..."

Baseline: injected text slips through

  • Retrieved assignment text contains an attack sentence
  • Baseline repeats it in the answer
  • No context sanitization
"Ignore all previous instructions..."

Guardrailed: context is sanitized

  • Retrieved context is treated as untrusted text
  • Instruction-like sentence is removed
  • Answer keeps the useful course evidence
answer cites assignment policy without the attack text

This is the main point of the milestone.

Without guardrails, the baseline RAG system retrieves whatever seems relevant. That means it can retrieve a private roster note if the question asks about it.

It can also repeat malicious instructions that are inside retrieved documents. This is indirect prompt injection: the user did not type the attack, but the retrieved context contains it.

The guardrailed version adds input checks, public-only retrieval filtering, and sanitization of retrieved text before answering.

Current limits

  • Local deterministic embeddings

  • Rule-based guardrails
  • Small 12-case evaluation set

prototype local demo ready for milestone

The evaluation is small, but it is useful for this milestone because it covers the required first failure modes: prompt injection, privacy, academic integrity, and unsupported questions.

The baseline passes only 3 out of 12 cases. The guardrailed version passes all 12 in this controlled test set.

The demo command runs the whole story: validates the corpus, rebuilds the vector index, runs normal course Q&A, writes the HTML visualization, then shows the baseline failures and the guardrailed behavior.

The honest limitation is that this is still a local deterministic prototype. It does not use production embeddings or a real LLM yet. That is next work.
